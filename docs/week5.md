# Week 5 요구사항 정리

## 목표

이번 주 과제의 목표는 React의 핵심 기능인 `Component`, `State`, `Hooks`를 직접 구현하고,
이를 기반으로 사용자 입력에 따라 실제로 화면이 바뀌는 웹 페이지를 만드는 것입니다.

또한 Week 4에서 구현한 `Virtual DOM`, `Diff`, `Patch` 흐름을 그대로 활용하여,
전체를 다시 그리지 않고 필요한 부분만 갱신하는 구조를 완성해야 합니다.

## 필수 구현 범위

### 1. Component

- UI를 작은 단위의 컴포넌트로 나누어 구현합니다.
- 각 컴포넌트는 자신의 렌더링 로직을 담당합니다.
- 반드시 함수형 컴포넌트로 구현합니다.

구현 방식:

- 함수형 컴포넌트를 감싸는 `FunctionComponent` 클래스를 직접 구현합니다.
- 이 클래스는 최소한 아래 기능을 포함해야 합니다.
  - `hooks` 배열: 훅 상태 저장
  - `mount()`: 최초 렌더링
  - `update()`: 상태 변경 후 다시 렌더링

### 2. State

- State는 변경 가능한 동적 데이터입니다.
- State가 바뀌면 화면이 자동으로 다시 렌더링되어야 합니다.
- State는 최상위 컴포넌트인 루트 컴포넌트에서만 관리합니다.

### 3. Hooks

함수형 컴포넌트에서도 상태와 부수효과를 다룰 수 있도록 최소한 아래 Hooks를 구현해야 합니다.

- `useState`
- `useEffect`
- `useMemo`

핵심 질문:

> 함수는 렌더링 때마다 다시 실행되는데, 상태는 어떻게 유지할 것인가?

이 과제에서는 이 문제를 `hooks` 배열과 렌더링 순서에 기반한 인덱스 관리 방식으로 해결해야 합니다.

### 4. Virtual DOM + Diff + Patch

Week 4에서 구현한 Virtual DOM 시스템을 그대로 활용합니다.

동작 흐름:

1. Virtual DOM 생성
2. 이전 Virtual DOM과 다음 Virtual DOM 비교
3. 변경된 부분만 실제 DOM에 반영

목표:

- 전체 DOM을 매번 다시 그리지 않습니다.
- 변경이 필요한 부분만 찾아서 갱신합니다.

### 5. 테스트 페이지

- 브라우저에서 실제로 동작하는 웹 페이지를 구현해야 합니다.
- 주제는 자유입니다.
- 반드시 사용자 입력, 클릭 등 상호작용에 따라 화면이 변경되어야 합니다.
- 즉, 구현한 `Component`, `State`, `Hooks`, `Virtual DOM` 흐름이 눈으로 확인되어야 합니다.

### 6. 기술 제한

- `JavaScript` 또는 `TypeScript`
- `HTML`, `CSS`
- 외부 프레임워크 사용 금지
  - 예: `React`, `Vue`, `Svelte` 등 사용 금지

## 제약사항

이번 과제의 가장 중요한 제약은 상태 관리 범위를 제한하는 것입니다.

- Hook은 최상위 컴포넌트에서만 사용 가능합니다.
- State는 루트 컴포넌트에서만 관리합니다.
- 자식 컴포넌트는 자체 State를 가지지 않습니다.
- 자식 컴포넌트는 부모로부터 받은 `props`만 사용합니다.
- 자식 컴포넌트는 상태가 없는 순수 함수형 컴포넌트로 구현합니다.

즉, 이번 과제는 `Lifting State Up` 패턴을 전제로 합니다.

## 구현 해석

요구사항을 구현 관점에서 정리하면 아래와 같습니다.

- 루트 컴포넌트가 모든 상태를 소유합니다.
- 루트 컴포넌트가 `useState`, `useEffect`, `useMemo`를 사용합니다.
- 자식 컴포넌트는 `props`를 받아 화면만 그립니다.
- 상태가 바뀌면 루트 컴포넌트가 다시 실행됩니다.
- 다시 실행된 결과로 새로운 Virtual DOM을 만듭니다.
- 이전 Virtual DOM과 diff를 계산합니다.
- patch를 적용해 실제 DOM을 최소 범위로 갱신합니다.

## 최소 성공 기준

아래 항목을 만족하면 이번 과제의 핵심 요구를 충족한 것으로 볼 수 있습니다.

- 함수형 컴포넌트 기반 구조가 동작한다.
- 루트 컴포넌트에 상태가 저장되고 유지된다.
- `useState`로 상태 변경 시 화면이 다시 그려진다.
- `useEffect`로 렌더링 이후 동작을 처리할 수 있다.
- `useMemo`로 계산 결과를 재사용할 수 있다.
- 자식 컴포넌트는 `props`만 받아 렌더링한다.
- 이전/다음 Virtual DOM을 비교하여 필요한 부분만 실제 DOM에 반영한다.
- 브라우저 데모에서 사용자 상호작용에 따라 UI가 바뀌는 것을 확인할 수 있다.

## 권장 구현 순서

1. `FunctionComponent` 클래스 구조를 정의합니다.
2. 루트 컴포넌트 기준으로 `hooks` 저장소와 hook index 흐름을 만듭니다.
3. `useState`를 먼저 구현하여 상태 변경과 재렌더링을 연결합니다.
4. `useEffect`, `useMemo`를 차례대로 추가합니다.
5. 컴포넌트 실행 결과를 Virtual DOM으로 만들고 렌더링합니다.
6. 상태 변경 시 이전/다음 Virtual DOM을 diff 하여 patch를 적용합니다.
7. 마지막으로 사용자 상호작용이 있는 테스트 페이지를 완성합니다.

## 한 줄 요약

이번 Week 5 과제는 `루트 컴포넌트에서만 상태와 훅을 관리하는 간단한 React 런타임`을 직접 만들고,
그 결과를 `Virtual DOM + Diff + Patch`로 화면에 반영하는 것입니다.
